Functional Test of the PART CPR Broker

This is a test of the CPR broker. It is mainly a black-box test using Soap UI UUIDs used for input in this test are created using CreateGUUID. 
Applicatíon1 token : 12BDB789-A8C0-452A-89E4-297F3583C3FB

How to perform the tests

This test is performed with the installation of the broker on the Gentofte network, http://cprbroker201100306 (application registration requests were performed on an earlier version of the broker).

Install SoapUI.

Create projects for the four webservices:

  • http://cprbroker2011306/Services/Admin.asmx?WSDL
  • http://cprbroker2011306/Services/Part.asmx?WSDL
  • http://eventbroker2011306/Services/Notification.asmx?WSDL
  • http://eventbroker2011306/Services/Subscriptions.asmx?WSDL
We also check the application page: http://cprbroker2011306/Pages/Applications.aspx

Fill in input for east test and check the output. The test are basicly black-box test that study mostly XML request and XML replies, a few of files (e.g. notification file channels to verify delivery of notifications), the web pages for administration, and a few queries on the database. We start with application registration because it is necessary to register "the test" as an application to perform the rest of the test. Notes


The RefreshRead replies using DPR and KMD differs slightly.

 
To show that ReadRefresh actually access global data, we would have to either update real CPR-data or update Gentoftes local DPR database and then do a new ReadRefresh query on the same UUID. Because this i live production data, we did not do this. This test gives a good indication that RefreshRead behaves as specified.



Test notes


The tests of application registration requests were performed on an
earlier version of the broker. They will not give the same results on
the latest version because the broker has been changed to require that
even registraions request supply a valid application token.
